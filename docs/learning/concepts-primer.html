CrewAI: Crews vs Flows vs Hierarchical

CrewAI is a Python framework for building multi-agent systems. It provides three orchestration patterns, and understanding which pattern to use when is the single most important CrewAI concept for P6.

First Principle

Every multi-agent system faces the same fundamental question: who decides what happens next? In a sequential Crew, the task list decides. In a hierarchical Crew, a manager agent decides (using an LLM). In a Flow, your code decides (using decorators and conditionals). The more deterministic your workflow, the more you want code-driven control (Flows). The more exploratory, the more you want agent-driven control (Crews).

PatternWho DecidesBranchingStateBest For
Sequential Crew The task list order None — linear only Output of task N feeds task N+1 Simple pipelines: research → write → edit
Hierarchical Crew A manager LLM agent Manager "chooses" (unreliably) Manager mediates Exploratory tasks (in theory — broken in practice)
Flow Your Python code via decorators Native: @router() Typed Pydantic FlowState Production systems with conditional logic, loops, shared state
Warning — Hierarchical is Broken

The hierarchical process is documented as non-functional in production (Towards Data Science, November 2025). The manager doesn't properly delegate — it runs tasks sequentially anyway but with extra LLM calls, causing loops, wrong agent invocation, and inflated costs. Do not use hierarchical for P6.

Java/Spring Parallel

A Crew is like a Spring @Service with multiple @Async methods — each agent does its job, and Spring manages execution. A Flow is like a Spring @EventListener chain — events fire sequentially, you control branching with if/else, and state is passed via typed event objects. The Flow gives you the control that @Async doesn't.

P6 Decision

P6 uses a Flow as the orchestration backbone (deterministic routing, typed state, conditional branching) with a single-agent Crew inside one Flow step for style-matched text generation (where LLM agency genuinely helps). This is CrewAI's own recommended production pattern.

02

CrewAI Flows Deep Dive

A Flow is a Python class that inherits from Flow[StateType]. Methods decorated with @start(), @listen(), and @router() define the execution graph. State is a Pydantic model that's automatically passed between steps.

The Three Decorators

@start() — The Entry Point

Marks the first method to run when flow.kickoff() is called. Every Flow must have exactly one @start() method. It initializes the state.

@start()
def begin(self):
    # This runs first. Set up initial state.
    self.state.query = "What is TCP/IP?"
    self.state.leader = "torvalds"

@listen(method_or_string) — "Run After This"

Marks a method to run after another method completes. You pass either a method reference (for chaining after a specific step) or a string (for routing from a @router).

@listen(begin)  # Runs after the 'begin' method
def retrieve_knowledge(self):
    self.state.chunks = rag_search(self.state.query)

@listen(retrieve_knowledge)  # Chains: begin → retrieve → this
def apply_style(self):
    self.state.response = generate_styled(self.state.chunks, self.state.leader)

@router() — Conditional Branching (The Key P6 Pattern)

A method decorated with both @listen() and @router() returns a string that routes to matching @listen("string") methods. This is how P6 implements deliver-vs-fallback.

@listen(apply_style)
@router()
def evaluate(self) -> str:
    score = compute_score(self.state.response)
    self.state.score = score
    if score >= 0.75:
        return "deliver"    # Routes to @listen("deliver")
    return "fallback"     # Routes to @listen("fallback")

@listen("deliver")  # Only runs if router returned "deliver"
def deliver_response(self):
    self.state.output = self.state.response

@listen("fallback")  # Only runs if router returned "fallback"
def handle_fallback(self):
    self.state.output = create_fallback(self.state)

FlowState — Typed Shared State

State is a Pydantic BaseModel. Every step reads from and writes to self.state. This is how data flows between steps without function arguments.

from pydantic import BaseModel

class CloneState(BaseModel):
    # Every step can read/write these fields
    query: str = ""
    leader: str = ""
    retrieved_chunks: list = []
    styled_response: str = ""
    evaluation: dict = {}
    final_output: dict = {}

class DigitalCloneFlow(Flow[CloneState]):
    # State is automatically created and managed
    # Access via self.state.query, self.state.leader, etc.
    ...
Java Parallel

FlowState is like a typed Context object passed through a chain-of-responsibility pattern. In Spring, imagine a HandlerChain where each handler reads from and writes to a shared RequestContext DTO. The Pydantic model gives you compile-time-like validation (at runtime) that a plain dict wouldn't.

Running a Flow

# Create and run
flow = DigitalCloneFlow()
result = flow.kickoff()  # Returns the output of the last step

# Access final state
print(flow.state.final_output)  # The full state is available after completion
Common Mistakes

1. The @router() method must return a string, not a boolean. return True won't route — use return "deliver".

2. @listen("deliver") matches the exact string returned by the router. Typos silently fail — the step just never runs.

3. Don't modify self.state from outside the Flow. All state changes happen inside decorated methods.

CrewAI Agents, Tasks & Crews

Even though P6 uses a Flow for orchestration, you still need a single-agent Crew for the style generation step. Here's how the building blocks work.

Agent — The Worker

An Agent has a role, goal, and backstory. These get injected into the LLM system prompt, guiding how the agent "thinks." For P6, the style agent's backstory includes the leader's communication patterns.

from crewai import Agent

style_agent = Agent(
    role="Communication style specialist for Linus Torvalds",
    goal="Generate responses matching Torvalds' writing style",
    backstory="You studied Torvalds' LKML emails. He uses short direct "
              "sentences, heavy dashes (--), starts with 'The point is', "
              "signs off with just 'Linus'. Formality: 0.45. Technical "
              "depth: 0.42.",
    llm="gpt-4o-mini",
    verbose=True
)

Task — The Job

A Task defines what an Agent should do. The description is the prompt, expected_output guides the format, and the task is assigned to a specific agent.

from crewai import Task

style_task = Task(
    description="Given these chunks: {chunks}\n"
                "Answer this query in Torvalds' style: {query}",
    expected_output="A response in Torvalds' communication style "
                    "with [N] citations",
    agent=style_agent
)

Crew — The Team

A Crew groups agents and tasks. In P6, the style Crew has exactly one agent and one task. kickoff(inputs={}) runs it, replacing {variable} placeholders in the task description.

from crewai import Crew

style_crew = Crew(
    agents=[style_agent],
    tasks=[style_task]
)

# Run with variable injection
result = style_crew.kickoff(inputs={
    "chunks": "[chunk text here]",
    "query": "What is TCP/IP?"
})

# Get the output
response_text = result.raw  # The raw string output
Why Only Style Uses a Crew

The RAGAgent step is deterministic (embed → FAISS search → rerank). No LLM "thinking" needed. The EvaluatorAgent is mostly math (cosine similarity + weighted formula). Only the ChatStyleAgent needs an LLM to generate text in a specific style, which benefits from CrewAI's role/goal/backstory prompt engineering. Using a Crew for deterministic steps adds overhead (extra LLM calls, slower execution) without benefit.

LiteLLM: Provider-Agnostic LLM Routing

LiteLLM wraps 100+ LLM providers (OpenAI, Anthropic, Cohere, Ollama, etc.) behind a single API. You call litellm.completion() and it routes to the right provider based on the model string. P5 already uses it — P6 continues the pattern.

First Principle

In production, you never want to be locked to one LLM provider. Pricing changes, rate limits hit, new models launch. LiteLLM is the adapter pattern: one interface, many implementations. If OpenAI raises prices tomorrow, you change one string ("gpt-4o-mini" → "claude-sonnet-4-20250514") and everything still works. This is the same principle as Java's JDBC — you code to the interface, not the driver.

Chat Completion (Text Generation)

from litellm import completion

# OpenAI
response = completion(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "Explain TCP/IP"}]
)
text = response.choices[0].message.content

# Switch to Anthropic — SAME interface, different model string
response = completion(
    model="claude-sonnet-4-20250514",
    messages=[{"role": "user", "content": "Explain TCP/IP"}]
)

Embeddings

from litellm import embedding

# OpenAI embeddings
response = embedding(
    model="text-embedding-3-small",
    input=["What is TCP/IP?"]
)
vector = response.data[0]["embedding"]  # list of 1536 floats

Integration with Instructor

LiteLLM + Instructor gives you validated Pydantic output from any LLM provider. This is your portfolio standard since P1.

import instructor
from litellm import completion

client = instructor.from_litellm(completion)

class StyleExplanation(BaseModel):
    explanation: str
    confidence: float

result = client.chat.completions.create(
    model="gpt-4o-mini",
    response_model=StyleExplanation,
    messages=[{"role": "user", "content": "Why is this response low confidence?"}]
)
# result is a validated StyleExplanation instance, not raw JSON
Common Gotchas

Model strings: "gpt-4o-mini" not "openai/gpt-4o-mini" (unless you're explicit-routing). For embeddings: "text-embedding-3-small" not "openai/text-embedding-3-small".

API keys: LiteLLM reads OPENAI_API_KEY from environment automatically. Set in .env file, loaded with python-dotenv.

Embedding response shape: response.data[0]["embedding"] — it's a list of dicts, each with an "embedding" key. Not response.embeddings.

Java Parallel

LiteLLM is like a RestTemplate that auto-configures based on the URL scheme. You call restTemplate.getForObject("openai://gpt-4o-mini", ...) vs restTemplate.getForObject("anthropic://claude-3", ...) — same method, different backends. Instructor is like Jackson's ObjectMapper.readValue() — it deserializes the LLM's text response into a typed Java object, with retries if the response doesn't match the schema.

Python mailbox: Parsing Email Archives

The mailbox module is part of Python's standard library. It reads mbox files — the format used by LKML archives on lore.kernel.org. Each message in the mbox is a Python email.message.Message object with headers and body.

import mailbox

# Open an mbox file
mbox = mailbox.mbox("data/emails/lkml.mbox")

for msg in mbox:
    sender = msg["From"]         # "Linus Torvalds <torvalds@linux-foundation.org>"
    subject = msg["Subject"]      # "Re: [PATCH] Fix null pointer dereference"
    date = msg["Date"]            # "Thu, 15 Sep 2022 08:23:45 -0700"
    message_id = msg["Message-ID"]

    # Get body text — handle multipart messages
    if msg.is_multipart():
        body = ""
        for part in msg.walk():
            if part.get_content_type() == "text/plain":
                payload = part.get_payload(decode=True)
                body = payload.decode("utf-8", errors="replace")
                break
    else:
        payload = msg.get_payload(decode=True)
        body = payload.decode("utf-8", errors="replace") if payload else ""

    # Filter: only Torvalds' emails
    if "torvalds@" in (sender or "").lower():
        print(f"Subject: {subject}, Words: {len(body.split())}")
LKML-Specific Issues

Encoding: LKML archives span decades. Expect ASCII, UTF-8, Latin-1, and broken encodings. Always use errors="replace" to avoid crashes.

Quoted text: LKML replies heavily quote previous messages with > prefixes. You MUST strip these or your style features will be contaminated by other people's writing.

Patches: Many Torvalds emails contain code patches (lines starting with +, -, @@). Strip these — they're not communication style, they're code.

Java Parallel

mailbox.mbox() is like javax.mail.internet.MimeMessage parsing, but simpler. In Java you'd use JavaMail with a Session and Store. Python's version is stateless — just open the file and iterate. msg["From"] is like mimeMessage.getFrom()[0].toString().

Style Feature Engineering

Style matching in P6 uses numerical features, not LLM embeddings. Each feature measures a quantifiable aspect of someone's writing. The 15 features form a vector in [0,1] space, and cosine similarity measures how closely a generated response matches the learned profile.

First Principle — Why Features, Not Embeddings

LLM embeddings capture semantic meaning — two texts about TCP/IP would have similar embeddings regardless of who wrote them. But style is about how you say things, not what you say. Torvalds and Kroah-Hartman might both explain TCP/IP correctly, but with completely different sentence lengths, punctuation patterns, and formality levels. Numerical features capture these surface-level patterns that embeddings deliberately ignore.

The 15 Features

FeatureHow to ComputeNormalization
Average message lengthsum(word_counts) / len(emails)Divide by max observed (e.g., 500 words)
Greeting patternsCount frequency of "Hi", "Dear", no greetingAlready proportions [0,1]
Punctuation patternsCount ..., !!, -- per emailDivide by max observed
Capitalization ratiouppercase_chars / total_charsAlready [0,1]
Question frequencycount("?") / len(emails)Divide by max observed
Vocabulary richnesslen(set(words)) / len(words)Already [0,1]
Common phrasesTF of top-10 phrases → encode as frequency vectorAlready proportions
Reasoning patternsFrequency of "because", "therefore", "however"Divide by total words
Sentiment distributionUse TextBlob or similar → pos/neu/neg ratiosAlready [0,1]
Formality levelFormal word count / total word countAlready [0,1]
Technical terminologyDomain term frequency (kernel, memory, syscall, etc.)Divide by total words
Code snippet freqEmails containing `backtick` or indented blocksCount / total emails
Quote-reply ratioLines starting with > / total lines (before stripping)Already [0,1]
Patch languageFrequency of "Applied", "NAK", "Acked-by"Count / total emails
Technical depthTechnical terms / general vocabularyAlready [0,1]

Cosine Similarity on Feature Vectors

Once you have a 15-dimensional vector for the leader's profile and another for the generated response, cosine similarity measures alignment:

import numpy as np

def style_score(profile_vec, response_vec):
    # Both are np.array of shape (15,), values in [0,1]
    dot = np.dot(profile_vec, response_vec)
    norms = np.linalg.norm(profile_vec) * np.linalg.norm(response_vec)
    return dot / norms if norms > 0 else 0.0

Target: > 0.90. If two profiles are identical, score = 1.0. If completely opposite, score = 0.0.

Incremental Learning with Weighted Averaging

Instead of recomputing the style profile from all 200+ emails every time a new batch arrives, you update incrementally using an exponential moving average (EMA).

# alpha = 0.3 means 30% weight on new data, 70% on existing profile
def update_profile(current: np.ndarray, new_batch: np.ndarray, alpha: float = 0.3):
    return (1 - alpha) * current + alpha * new_batch

Why this matters: in production, emails arrive continuously. You can't re-process 10,000 emails every time 5 new ones come in. The alpha parameter controls how quickly the profile adapts — high alpha (0.7) = rapid adaptation, low alpha (0.1) = very stable profile.

Java Parallel

This is the same pattern as a MovingAverage in a trading system or a ExponentiallyDecayingReservoir in Dropwizard Metrics. Each new data point contributes proportionally, and old data gradually fades. The alpha parameter is the "decay factor."

Multi-Agent Topologies

Multi-agent systems arrange agents in different patterns depending on the problem structure. Understanding these patterns is an interview differentiator — it shows you think about system design, not just code.

TopologyHow It WorksWhen to UseP6 Relevance
Sequential (Pipeline) A → B → C. Output of each feeds the next. Well-defined stages. ETL, document processing. P6's main flow: RAG → Style → Evaluate
Parallel (Fan-out/Fan-in) A → [B, C] → D. Multiple agents work simultaneously, results merged. Independent subtasks. Parallel search, multi-model comparison. Dual-leader mode: same RAG → [Torvalds style, GKH style] → Compare
Hierarchical (Manager-Worker) Manager delegates to workers based on task type. Dynamic routing. Customer support triage. Not used — P6's routing is deterministic, not LLM-decided.
Reflexive (Self-Critique) Agent evaluates its own output and iterates. Quality improvement loops. Code review agents. EvaluatorAgent critiques StyleAgent's output (one-shot, not iterative).
Interview Signal

"P6 uses a sequential pipeline with a conditional branch and a parallel fan-out for dual-leader comparison. I chose sequential because the workflow is deterministic — RAG always runs before style, style always runs before evaluation. The only non-deterministic decision is deliver-vs-fallback, which is handled by a Flow @router, not by giving an LLM the choice. For dual-leader mode, I fan out after shared RAG retrieval, run two parallel style+evaluate pipelines, and fan in for the comparison view."
